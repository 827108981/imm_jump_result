# 手机热点 H5 采集端需求说明

版本：V0.1  
状态：待评审  
适用项目：免疫跳值排查反馈报告生成工具  
核心目标：让一线工程师用手机在现场直接拍照、填写记录，并自动绑定到 PC 工具中的具体排查步骤，减少“手机拍照 -> 传电脑 -> 找步骤 -> 再上传”的重复操作。

## 1. 背景

当前一线排查流程中，工程师通常需要使用手机拍照，再通过微信、数据线、网盘或其他方式传到电脑，随后在 PC 工具中逐项上传照片和填写记录。该流程存在以下问题：

- 照片和排查步骤没有天然绑定，后续整理容易错放、漏放。
- 手机到电脑的文件传输耗时，且依赖现场网络或第三方工具。
- 第 35 步拆分后存在多个子项，照片/记录要求不同，人工填写更容易遗漏。
- RTS 审核发现资料不全后，一线需要重新翻完整报告定位补充项。

本需求建议第一阶段先采用“手机热点 + PC 本地服务 + 手机 H5 页面”的方式实现移动采集，不开发 iOS 原生 App。

## 2. 总体方案

现场由手机开启热点，电脑连接该手机热点。PC 工具启动后监听局域网地址并生成二维码，手机扫码打开 H5 采集页面。

手机端 H5 页面展示第 1 步到第 39 项采集项，一线工程师可直接在手机上按步骤拍照、填写记录、选择结论。所有数据实时上传到当前电脑，由 PC 工具统一生成报告并返回给 RTS。

一句话目标：把照片从“普通图片文件”变成“带步骤身份的数据”。

## 3. 网络与部署条件

### 3.1 推荐现场连接方式

1. 工程师手机开启个人热点。
2. 电脑连接该手机热点。
3. PC 工具启动本地服务。
4. PC 页面显示手机采集二维码。
5. 手机扫码进入采集页面。

### 3.2 PC 工具要求

- Flask 服务需要监听 `0.0.0.0`，不能只监听 `127.0.0.1`。
- PC 工具需要自动识别电脑在手机热点网络中的 IP，例如 `172.20.10.2` 或 `192.168.x.x`。
- 二维码 URL 必须使用电脑局域网 IP，例如：

```text
http://172.20.10.2:5000/mobile?task_id=xxx&token=yyy
```

- Windows 防火墙需允许该工具在“专用网络”中被访问。
- PC 页面应提供“连接测试”状态，提示手机是否已成功访问。

### 3.3 手机要求

- Android：可使用 Android App 或浏览器 H5。
- iPhone：使用 Safari 打开 H5 页面，不需要安装 iOS App。
- 手机浏览器需支持图片选择/拍照上传。
- 第一阶段不要求 HTTPS，不要求 App Store，不要求云服务器。

### 3.4 需实机验证的网络点

不同手机热点策略可能存在差异，第一版开发前需验证：

- iPhone 开热点后，iPhone Safari 是否能访问连接到热点的 Windows 电脑 IP。
- Android 开热点后，Android 浏览器是否能访问连接到热点的 Windows 电脑 IP。
- 公司电脑安全策略是否拦截入站端口。

如手机热点无法从手机访问电脑，可保留备用方案：电脑开启热点、便携路由器、或云中转。

## 4. 阶段规划

## 第一阶段：手机热点 H5 采集端

目标：不开发 iOS 原生 App，通过手机浏览器完成现场拍照、记录填写和上传。

范围：

- PC 端生成手机采集任务和二维码。
- 手机 H5 展示第 1 步到第 39 项采集项。
- 手机端支持拍照/选择图片上传。
- 手机端支持填写实测记录、选择排查结论。
- 数据自动同步到 PC 当前任务。
- PC 端可继续检查完整性并生成报告。

不包含：

- 离线缓存。
- 断点续传。
- 语音转文字。
- 照片水印。
- RTS 补充项回流。
- iOS 原生 App。

## 第二阶段：Android App 增强采集端

目标：在 Android App 中提供更强的现场采集体验。

增强能力：

- 离线缓存。
- 断点续传。
- 本地照片压缩。
- 照片自动水印：医院、设备序列号、步骤编号、时间。
- 语音转文字填写记录。
- 扫码绑定 PC 当前任务。
- 上传失败自动重试。
- 批量拍照后按步骤归档。

## 第三阶段：RTS 审核闭环

目标：RTS 审核后生成结构化补充清单，一线只需补充指定项。

能力：

- RTS 在审核报告中标记需补充项。
- 系统生成补充清单。
- 一线扫码后只看到需补充的步骤。
- 补充完成后自动合并到原报告数据。
- RTS 可对补充后的报告再次审核。

## 5. 第一阶段详细需求

## 5.1 PC 端功能

### 5.1.1 手机采集入口

PC 首页新增“手机采集”区域，显示：

- 当前电脑可访问 IP。
- 当前服务端口。
- 手机采集二维码。
- 连接状态：未连接 / 已连接 / 最近同步时间。
- 操作按钮：刷新二维码、复制链接、关闭手机采集。

### 5.1.2 任务创建

PC 工具启动或用户点击“开启手机采集”时创建一个采集任务。

任务信息包括：

- `task_id`：采集任务 ID。
- `session_id`：当前报告会话 ID。
- `token`：一次性访问令牌。
- `created_at`：创建时间。
- `expires_at`：过期时间，默认 8 小时。
- `status`：active / closed / expired。

### 5.1.3 二维码内容

二维码内容为手机 H5 地址：

```text
http://{pc_lan_ip}:{port}/mobile?task_id={task_id}&token={token}
```

要求：

- URL 中必须包含 `task_id` 和 `token`。
- token 至少 24 位随机字符。
- PC 端可刷新 token，旧 token 立即失效。

### 5.1.4 PC 端实时同步

PC 页面应能看到手机端上传的数据。

第一阶段可使用轮询：

- 手机端上传后写入当前 session 数据。
- PC 端每 2 秒拉取一次最新数据。
- 如果 PC 端已有未保存内容，需要做合并，不允许手机同步直接覆盖 PC 已填写内容。

推荐合并规则：

- 同一字段 PC 为空、手机有值：自动填入。
- 同一字段 PC 有值、手机为空：保留 PC。
- 同一字段 PC 和手机都有值且不同：保留 PC，并提示“手机端有更新待确认”。
- 图片上传：追加到对应图片列表，不覆盖已有图片。

### 5.1.5 完整性检查

现有完整性检查继续由 PC 端负责。

新增要求：

- 手机上传的数据必须参与 PC 端完整性检查。
- 未填写项继续红框定位。
- 第 35 步规则按当前版本执行：
  - 第35步-1 本底测试：必须上传调试或维护后照片。
  - 第35步-2 样本针和试剂针加样重复性测试：必须上传调试或维护后照片。
  - 第35步-3 清洗（缓冲）液检测：必须上传调试或维护后照片。
  - 第35步-4 低值质控重复性测试：必须填写测试记录。

## 5.2 手机 H5 功能

### 5.2.1 打开页面

手机扫码进入 H5 页面后，系统校验：

- `task_id` 是否存在。
- `token` 是否有效。
- 任务是否过期。
- PC 服务是否可访问。

异常时显示明确提示：

- “任务不存在，请在电脑端重新生成二维码。”
- “二维码已过期，请刷新二维码。”
- “无法连接电脑，请确认电脑已连接手机热点，并允许防火墙访问。”

### 5.2.2 任务首页

手机端首页展示：

- 医院名称、设备型号、设备序列号、跳值项目。
- 当前完成进度。
- 排查项列表：第 1 步到第 39 项。
- 每项状态：未开始 / 已填写 / 缺照片 / 缺记录 / 已完成。
- 快速筛选：全部、未完成、缺照片、缺记录、异常/待确认。

### 5.2.3 步骤详情页

每个排查项详情展示：

- 步骤编号。
- 排查动作。
- 合格指标。
- 必填要求。
- 实测情况记录输入框。
- 排查结论选择：正常 / 异常 / 已处理 / 待确认。
- 原始状态照片上传区。
- 调试或维护后照片上传区。
- 保存并下一项。

### 5.2.4 拍照与上传

手机端使用浏览器文件上传能力：

```html
<input type="file" accept="image/*" capture="environment">
```

要求：

- iPhone Safari 可选择拍照或从相册选择。
- Android 浏览器可选择拍照或从相册选择。
- 支持单张或多张上传。
- 每个上传区最多 5 张。
- 支持 jpg、jpeg、png、webp。
- 上传前显示缩略图和上传状态。
- 上传失败可重试。

图片压缩：

- 第一阶段可沿用 PC 端压缩逻辑。
- 手机端可先不做本地压缩。
- 如上传速度过慢，第二阶段再加入手机端压缩。

### 5.2.5 保存记录

手机端填写记录后应支持：

- 自动保存。
- 手动保存。
- 保存并下一项。

建议第一阶段：

- 输入框失焦后自动保存。
- 点击“下一项”前强制保存。
- 页面顶部显示保存状态：已保存 / 保存中 / 保存失败。

### 5.2.6 防误操作

要求：

- 手机端不允许删除 PC 端已有图片，只允许删除本手机本次上传的图片。
- 删除图片前二次确认。
- 返回列表时保留当前填写内容。
- 网络中断时提示“未同步，请稍后重试”。

## 7. 接口草案

### 7.1 PC 创建采集任务

```http
POST /api/mobile/task/create
```

返回：

```json
{
  "ok": true,
  "task_id": "m_xxx",
  "mobile_url": "http://172.20.10.2:5000/mobile?task_id=m_xxx&token=xxx",
  "qr_url": "/api/mobile/task/m_xxx/qrcode"
}
```

### 7.2 手机获取任务

```http
GET /api/mobile/task?task_id=m_xxx&token=xxx
```

返回：

```json
{
  "ok": true,
  "base_info": {},
  "items": [],
  "item_data": {}
}
```

### 7.3 手机保存记录

```http
POST /api/mobile/item/save
```

请求：

```json
{
  "task_id": "m_xxx",
  "token": "xxx",
  "item_id": "item_035_4",
  "measured_value": "低值质控重复性测试记录...",
  "conclusion": "正常"
}
```

### 7.4 手机上传图片

```http
POST /api/mobile/upload
```

表单字段：

- `task_id`
- `token`
- `item_id`
- `field`：before_images / after_images
- `files`

### 7.5 PC 拉取同步状态

```http
GET /api/mobile/task/status?task_id=m_xxx
```

返回：

```json
{
  "ok": true,
  "connected": true,
  "last_seen_at": "2026-07-08 15:40:00",
  "updated_item_ids": ["item_035_2"]
}
```

## 8. 安全要求

第一阶段不走云端，数据只在手机和当前电脑之间传输。

仍需满足：

- 手机采集 URL 必须带随机 token。
- token 过期后不可访问。
- PC 端可手动关闭采集任务。
- 上传接口必须校验 task_id 和 token。
- 限制图片类型和大小。
- 限制单项图片数量。
- 不允许访问 uploads 目录之外的文件。
- 日志中不记录完整 token，只记录 task_id 和末尾 4 位。

## 9. 兼容性要求

### 9.1 浏览器

第一阶段目标浏览器：

- iPhone Safari。
- Android Chrome。
- Android 系统浏览器作为兼容目标。

### 9.2 页面适配

- 手机页面优先适配 360px 到 430px 宽度。
- 按钮高度不低于 40px。
- 上传按钮、保存按钮、下一项按钮必须适合单手点击。
- 页面避免复杂表格。

### 9.3 弱网处理

- 上传中显示进度或等待状态。
- 请求超时后允许重试。
- 保存失败不清空输入内容。
- 页面顶部显示网络/连接状态。

## 10. 验收标准

### 10.1 连接验收

- 手机开热点，电脑连接热点后，PC 页面可生成二维码。
- iPhone Safari 扫码可打开 H5 页面。
- Android 浏览器扫码可打开 H5 页面。
- 手机端访问失败时有明确提示。

### 10.2 采集验收

- 手机端能看到第 1 步到第 39 项。
- 手机端能进入任一排查项详情。
- 手机端能填写实测记录并同步到 PC。
- 手机端能上传原始状态照片并同步到 PC。
- 手机端能上传调试或维护后照片并同步到 PC。
- PC 端完整性检查能识别手机端上传的数据。

### 10.3 第 35 步验收

- 第35步-1 本底测试：未上传调试或维护后照片时，完整性检查标红。
- 第35步-2 样本针和试剂针加样重复性测试：未上传调试或维护后照片时，完整性检查标红。
- 第35步-3 清洗（缓冲）液检测：未上传调试或维护后照片时，完整性检查标红。
- 第35步-4 低值质控重复性测试：未填写测试记录时，完整性检查标红。

### 10.4 报告验收

- 手机端采集的数据能进入最终 HTML 报告。
- 手机端上传的图片能进入报告 images 目录。
- report_data.json 中能保留对应 item_id 和图片路径。
- 生成 ZIP 包后 RTS 可正常审核。

## 11. 风险与待确认事项

### 11.1 手机热点连通性

风险：部分手机热点或电脑安全策略可能阻止手机访问电脑服务。

处理：

- 开发前做 iPhone 热点、Android 热点实机连通性验证。
- PC 端提供连接测试页面。
- 保留备用网络方案。

### 11.2 Windows 防火墙

风险：第一次运行 exe 时，Windows 防火墙可能阻止手机访问。

处理：

- PC 页面提示用户允许“专用网络”访问。
- 提供“手机无法打开二维码页面”的排查提示。

### 11.3 iOS Safari 上传体验

风险：iOS 不同版本拍照上传交互略有差异。

处理：

- 使用标准 file input。
- 不依赖 getUserMedia。
- 不做必须 HTTPS 才可用的高级摄像头能力。

### 11.4 多端冲突

风险：PC 和手机同时编辑同一项，产生覆盖。

处理：

- 第一阶段采用保守合并策略。
- PC 有值时不被手机空值覆盖。
- 冲突时提示用户确认。

## 12. 建议开发里程碑

### M1：连通性原型

- PC 监听 `0.0.0.0`。
- 自动获取局域网 IP。
- PC 显示手机访问二维码。
- 手机打开 H5 页面并显示步骤列表。

### M2：基础采集闭环

- 手机端填写记录。
- 手机端上传图片。
- PC 端同步显示。
- 完整性检查读取手机端数据。

### M3：体验优化

- 手机端进度状态。
- 自动保存。
- 上传失败重试。
- 连接状态提示。

### M4：打包与现场试点

- 打包 exe。
- Android 热点测试。
- iPhone 热点测试。
- 现场模拟完整排查。
- 输出问题清单和下一版优化项。

## 13. 第一版推荐结论

第一版建议只做“手机热点 + H5 采集 + PC 同步 + 报告生成”。

不要一开始做云端、iOS App、离线缓存和 RTS 回流。先把现场最高频、最耗时的照片绑定问题解决，能最快提升一线效率。
