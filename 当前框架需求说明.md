# 免疫跳值排查反馈报告生成工具：当前框架需求说明

版本：V1.7  
状态：当前正式需求基线  
更新日期：2026-07-21

## 1. 目标与范围

本工具用于完成免疫跳值排查的闭环：一线采集和补充、RTS/GTS 审核、审核返回、再次补充并重新导出。

正式交付包含两种填写入口：

1. **电脑端程序**：用于一线填写、RTS/GTS 审核及报告导出。
2. **手机离线单文件**：用于一线在无电脑、无网络的现场采集、拍照、保存、返修和导出。

RTS 与 GTS 使用同一审核流程，界面、文案和返回文件统一称为“RTS/GTS”。

## 2. 闭环流程

1. 一线在电脑端或手机离线端填写基础信息、39 个排查步骤、记录和照片。
2. 一线执行完整性检查，通过后导出正式一线 ZIP，并发送给 RTS/GTS。
3. RTS/GTS 在电脑审核界面导入一线 ZIP，逐项选择“通过”或“需补充”，填写审核意见与补充要求后，导出 RTS/GTS 返回 ZIP。
4. 一线收到 RTS/GTS 返回 ZIP 后，**直接导入这一份返回 ZIP**。系统恢复原报告、原照片和补充清单，不需要再先导入原一线 ZIP。
5. 一线仅补充 RTS/GTS 指定内容，确认补充、重新通过完整性检查后，导出新的正式一线 ZIP，发送 RTS/GTS 复核。
6. 如仍需补充，重复第 3 至第 5 步，直到审核通过。

所有入口均需醒目展示红色提示：**“使用前务必读懂闭环流程（使用须知）”**；“闭环流程”在页面内统一命名为“使用须知”。

## 3. 一线电脑端

### 3.1 功能要求

- 读取系统内置的 39 项排查模板，展示基础信息、步骤记录、原始状态照片和调试/维护后照片。
- 根据模板字段规则校验必填、选填、记录和照片要求。
- 支持单步骤的排查结论、异常/已处理/待确认汇总，以及问题编号展示。
- 支持保存、加载和覆盖电脑端草稿。
- 支持图片预览与旋转查看；不对照片自动添加水印。
- 支持完整性检查，定位未填项、缺记录项、缺照片项和未完成的 RTS/GTS 补充项。
- 导出正式一线 ZIP 前必须通过完整性检查。
- 页面仅提供“导入 RTS/GTS 返回 ZIP”，不再要求或引导用户先导入原一线 ZIP。

### 3.2 RTS/GTS 返回 ZIP 导入

导入 RTS/GTS 返回 ZIP 后，电脑端必须：

- 还原基础信息、全部步骤记录、结论和照片，恢复为可编辑状态。
- 读取审核信息和逐项补充要求。
- 高亮需要补充的步骤，并支持只查看补充项。
- 在补充项未完成时阻止正式导出。
- 一线完成补充后允许重新检查并再次导出正式一线 ZIP。

## 4. 手机离线单文件

### 4.1 交付与打开方式

- 文件名为 `mobile_offline.html`，为可独立分发的单个 HTML 文件。
- 必须用 Chrome、Edge 或较新的 Safari 浏览器打开；微信内置浏览器仅作为文件传递渠道，不作为长时间填写环境。
- 不依赖电脑、局域网、服务器或外网；打开本地文件后可完全离线填写。
- 手机端和电脑端使用同一份模板、相同的字段规则、步骤规则、完整性规则和正式 ZIP 格式。

### 4.2 本机保存与可靠性

- 报告数据与图片保存在当前浏览器的 IndexedDB 中，并在填写、拍照、删除照片、导入返回 ZIP、确认补充等动作后自动保存。
- 支持新建、切换、继续和删除多份本机报告。
- 页面切到后台、锁屏或短暂退出后，重新打开同一浏览器中的同一 HTML 时，应可继续本机已保存的报告。
- 浏览器缓存被用户主动清除、使用无痕模式或更换浏览器/设备后，不能保证保留本机草稿；现场应定期导出草稿 ZIP 备份。

### 4.3 图片与容量

- 支持拍照和从相册选择图片。
- 支持 `jpg`、`jpeg`、`png`、`webp` 等常见图片格式。
- 每个图片字段最多 5 张照片。
- 正式 ZIP 导入和导出上限均为 500 MB；超过上限时给出明确提示，要求减少或压缩照片。
- 不自动给照片加水印。

### 4.4 返回 ZIP 与再次导出

- 手机端只提供“导入 RTS/GTS 返回 ZIP”，不提供原一线 ZIP 导入入口。
- 返回 ZIP 中必须包含原报告、原照片、RTS/GTS 审核意见和补充要求；导入后自动创建或更新本机可编辑报告。
- 导入后展示每项补充要求，允许一线确认该步骤已按要求补齐。
- 未确认全部补充要求前，正式 ZIP 导出必须被阻止。
- 支持导出草稿 ZIP；草稿用于备份或继续填写，正式 ZIP 仅在完整性检查通过后生成。
- 正式 ZIP 命名规则：`医院*机型*序列号_yyyyMMddHHmmss.zip`。文件名中的不安全字符需自动替换。

### 4.5 手机界面要求

- 默认适配常见竖屏手机宽度，不出现需要横向滑动才能看到主要操作按钮或填写内容的情况。
- “更多”菜单打开后，点击菜单外区域必须关闭。
- 主要操作包括：切换报告、新建、导入 RTS/GTS 返回 ZIP、导出正式 ZIP、检查完整性和查看使用须知。
- 手机端报告中的图片查看需支持旋转。

## 5. RTS/GTS 审核端

### 5.1 导入与审核

- 审核端导入一线正式 ZIP；审核返回 ZIP 应保留一线报告数据和重点图片，以支持一线直接返修导入。
- 每个步骤均必须有明确审核决定：`通过` 或 `需补充`，不得出现无法理解的占位值（如 `??`）。
- 支持“一键全部通过”，随后仍可逐项改为“需补充”。
- 对“需补充”的步骤可选择：补记录、补原始照片、补调试后照片，并填写具体补充要求。
- 审核端必须填写审核结论和审核说明，校验所有步骤已完成审核后才可生成返回 ZIP。
- 审核页面需要清晰展示完整问题编号，不能被卡片宽度裁切。

### 5.2 审核返回 ZIP

RTS/GTS 返回 ZIP 至少应包含：

- `rts_review_data.json`：审核结论、逐项审核决定、补充要求和原一线数据。
- 原一线报告所需的图片文件，路径与数据引用一致。
- 审核报告 HTML（用于阅读和归档）。

该 ZIP 是一线返修的唯一导入文件；一线无需同时保留或再次导入最初的一线 ZIP。

## 6. ZIP 数据契约

### 6.1 一线正式 ZIP

正式一线 ZIP 必须包含：

- `report_data.json`：基础信息、模板版本、步骤记录、照片引用、完整性结果和 RTS/GTS 补充完成状态。
- 报告 HTML。
- 图片目录及实际图片文件。
- 机器可读的清单/元数据，用于导入和完整性追踪。

### 6.2 格式与安全规则

- 当前包格式版本为 `frontline-report-v2`。
- 导入时必须校验 ZIP 结构、文件路径、条目数量、图片数量和 500 MB 容量限制，拒绝不符合格式或存在路径越界风险的文件。
- RTS/GTS 返回 ZIP 中的原始数据与照片必须可被电脑端和手机端共同读取。
- 导出前的完整性结果必须写入 ZIP；正式包不能以“草稿/未完成”状态导出。

## 7. 当前正式入口与维护边界

### 7.1 正式入口

- 电脑一线填写页：`/`
- RTS/GTS 审核页：`/rts`
- 手机离线文件预览/下载：`/mobile-offline.html`、`/mobile-offline/download`
- 发布目录：`release/`
  - `免疫跳值排查反馈报告生成工具.zip`
  - `手机离线填写.zip`

后续发布应直接覆盖 `release/` 下同名程序包和手机离线包，不保留旧版本发布包、旧输出目录或旧生成模板。

### 7.2 非正式/历史实现

项目中可能仍存在早期的手机热点实时同步页面、旧报告模板或旧构建脚本。它们不属于当前正式闭环，不应继续增加功能，也不应作为一线交付入口。若删除这些历史实现，必须先确认不会影响上述三个正式入口及 ZIP 导入导出契约。

## 8. 验收标准

以下场景必须全部通过：

1. 电脑端一线填写完成后，完整性检查可准确拦截缺失内容；通过后能导出正式 ZIP。
2. RTS/GTS 导入一线 ZIP，可对全部 39 项作出“通过/需补充”决定，并生成返回 ZIP。
3. 电脑端只导入 RTS/GTS 返回 ZIP，即可恢复原基础信息、步骤记录、照片及补充项，并完成返修后重新导出。
4. 手机端在断网状态下填写、拍照、切后台、重新打开后可继续同一份本机报告。
5. 手机端只导入 RTS/GTS 返回 ZIP，即可恢复原报告、照片和补充项；未确认补充项时不能正式导出。
6. 手机常见竖屏宽度下，主要按钮和表单不产生横向溢出；“更多”菜单可通过点击外部关闭。
7. 两端均可生成符合命名规则且小于等于 500 MB 的正式 ZIP；两端生成的 ZIP 可被 RTS/GTS 审核端正确导入。
8. 电脑端和手机端的图片查看均支持旋转，且照片不带自动水印。
